## 一、现状

### 1.1 两套视觉节点

Pi5 上 `my_srv` 包中存在两套视觉节点，功能一一对应：

| 功能 | 2D 单目版本 | 深度版本 |
|------|-----------|---------|
| 颜色分拣 | `/color_sorting/enter` `/color_sorting/exit` | `/depth_color_sorting/enter` `/depth_color_sorting/exit` |
| 颜色码垛 | `/color_stack/enter` `/color_stack/exit` | `/depth_color_stack/enter` `/depth_color_stack/exit` |
| 颜色追踪 | `/color_track/enter` `/color_track/exit` | `/depth_color_track/enter` `/depth_color_track/exit` |
| 标签分拣 | `/label_sorting/enter` `/label_sorting/exit` | `/depth_label_sorting/enter` `/depth_label_sorting/exit` |
| 标签码垛 | `/label_stack/enter` `/label_stack/exit` | `/depth_label_stack/enter` `/depth_label_stack/exit` |
| 标签追踪 | `/label_track/enter` `/label_track/exit` | `/depth_label_track/enter` `/depth_label_track/exit` |
| 数字分拣 | `/num_sorting/enter` `/num_sorting/exit` | `/depth_num_sorting/enter` `/depth_num_sorting/exit` |
| 数字码垛 | `/num_stack/enter` `/num_stack/exit` | `/depth_num_stack/enter` `/depth_num_stack/exit` |
| 数字追踪 | `/num_track/enter` `/num_track/exit` | `/depth_num_track/enter` `/depth_num_track/exit` |

**规律：深度版本 = `/depth_` + 2D 版本路径**

### 1.2 当前离线语音节点的问题

`asr_node.py` 只处理 2D 版本的命令（如 `color_sorting_enter`），没有深度版本的映射。

### 1.3 USB 设备识别

当前 Pi5 上的 USB 设备：

```
Bus 004 Device 004: ID 3251:1930  Aurora 930 深度相机
Bus 002 Device 002: ID 4c4a:4155  UACDemoV1.0 (USB 声卡)
```

- **Aurora 930 深度相机**：USB VID:PID = `3251:1930`
- **普通 USB 单目相机**：标准 UVC 设备，会出现在 `/dev/video*`，VID:PID 因品牌而异

## 二、相机检测方案

### 2.1 检测方法

```python
import subprocess

def detect_camera():
    """检测相机类型
    返回: "depth" | "mono" | "both" | "none"
    """
    result = subprocess.run(['lsusb'], capture_output=True, text=True)
    lsusb = result.stdout

    has_depth = '3251:1930' in lsusb  # Aurora 930
    has_mono = False

    # 检查 /dev/video* 是否有 UVC 摄像头（排除 Pi5 自带的 pispbe/rpivid）
    import glob
    for dev in glob.glob('/dev/video*'):
        dev_num = int(dev.replace('/dev/video', ''))
        if dev_num < 19:  # video0-18 可用，video19+ 是 Pi5 硬件编解码器
            has_mono = True
            break

    if has_depth:
        return "depth"  # 有深度相机就用深度（无论是否有单目）
    elif has_mono:
        return "mono"
    else:
        return "none"
```

### 2.2 检测时机

- **启动时检测一次**：节点启动时判断，结果作为整个会话的模式
- **不支持运行时热切换**：拔插相机需要重启节点（简洁优先原则）

### 2.3 判定逻辑

```
有深度相机 → 深度模式（无论是否有单目）
无深度相机 + 有单目 → 单目模式
都没有 → 报错退出
```

## 三、节点路由方案

### 3.1 核心思路

**LLM 输出的 function 名不变，执行器节点根据相机模式自动拼接 ROS2 路径前缀。**

```
LLM 输出: {"function": "color_sorting", "parameters": {"action": "enter"}}

深度模式 → 调用 /depth_color_sorting/enter
单目模式 → 调用 /color_sorting/enter
```

### 3.2 执行器节点路由逻辑

```python
class ArmExecutorNode(Node):
    def __init__(self):
        # 启动时检测相机
        self.camera_mode = detect_camera()  # "depth" or "mono"
        # 深度模式下路径前缀
        self.prefix = "depth_" if self.camera_mode == "depth" else ""

    def execute_step(self, step):
        func = step["function"]
        params = step.get("parameters", {})

        if func == "color_sorting":
            if params["action"] == "enter":
                self.call_service(f'/{self.prefix}color_sorting/enter')
            elif params["action"] == "exit":
                self.call_service(f'/{self.prefix}color_sorting/exit')
            elif params["action"] == "sort":
                self.publish(f'/{self.prefix}color_sorting/command', ...)

        # 其他功能同理...
```

## 四、SYSTEM_PROMPT 设计

**LLM 不需要知道相机类型。** SYSTEM_PROMPT 中的功能定义保持统一，执行器节点在底层自动路由。

```
用户说: "帮我把红色的挑出来"
    ↓
LLM 输出: {"function": "color_sorting", "parameters": {"action": "sort", "colors": ["red"]}}
    ↓
执行器节点:
  深度模式 → publish /depth_color_sorting/command "sort red"
  单目模式 → publish /color_sorting/command "sort red"
```

这样做的好处：
- **SYSTEM_PROMPT 只写一份**，不区分深度/单目
- **路由逻辑集中在执行器节点**，改一处即可
- **LLM 不增加复杂度**，理解能力不受影响

| 问题 | 方案 |

| 如何检测相机类型 | `lsusb` 检查 VID:PID `3251:1930` (Aurora 930) |

| 判定逻辑 | 有深度→深度模式；只有单目→单目模式；都没有→报错 |
| 路由方式 | 执行器节点根据模式自动拼接 `/depth_` 前缀 |
| SYSTEM_PROMPT | 只写一份，不区分相机类型 |
