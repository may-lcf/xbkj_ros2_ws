# 在线语音交互实施方案分析

> 分析日期：2026-06-08
> 目标：在现有离线语音交互基础上，增加在线语音交互能力
> 要求：非固定命令词，语义相同即可命中（更灵活）

---

## 一、离线语音交互系统分析

### 1.1 硬件组成

```
天问 ASRPRO 语音识别模块
├── 内置 DSP：离线命令词识别（不依赖网络）
├── 麦克风：采集语音
└── UART 串口：识别到命令词后发送对应字符串
    └── /dev/ttyUSB0 (115200bps)
```

### 1.2 工作流程

```
用户说 "开启颜色分拣"
    │
    ▼
ASRPRO 硬件匹配命令词 "开启颜色分拣"
    │
    ▼
串口发送字符串 "color_sorting_enter\n"
    │
    ▼
asr_node.py receive_loop() 读取串口
    │
    ▼
elif command_str == "color_sorting_enter":
    │
    ▼
调用 ROS2 服务 /color_sorting/enter
    │
    ▼
color_sorting_node.py 启动颜色分拣
```

### 1.3 命令词 → 字符串 → ROS2 动作 映射表

asr_node.py 中硬编码了约 40 条命令词映射：

| ASRPRO 命令词 | 串口字符串 | ROS2 动作 |
|--------------|-----------|----------|
| 开启颜色分拣 | `color_sorting_enter` | 调用 `/color_sorting/enter` 服务 |
| 关闭颜色分拣 | `sort stop color` | 调用 `/color_sorting/exit` 服务 |
| 分拣红色 | `sort red` | 发布到 `/color_sorting/command` |
| 分拣红绿蓝 | `sort red green blue` | 发布到 `/color_sorting/command` |
| 开启颜色码垛 | `color_stack_enter` | 调用 `/color_stack/enter` 服务 |
| 开启颜色追踪 | `track on` | 调用 `/color_track/enter` 服务 |
| 追踪红色 | `track red` | 发布到 `/color` topic |
| 追踪绿色 | `track green` | 发布到 `/color` topic |
| 追踪蓝色 | `track blue` | 发布到 `/color` topic |
| 开启标签分拣 | `label_sorting_enter` | 调用 `/label_sorting/enter` 服务 |
| 分拣标签1 | `sort_1` | 发布到 `/label_sorting/sort_command` |
| 开启数字分拣 | `num_sorting_enter` | 调用 `/num_sorting/enter` 服务 |
| 开启人脸追踪 | `face_track_on` | 调用 `/face_track/enter` 服务 |
| 开启标签追踪 | `label_track_on` | 调用 `/label_track/enter` 服务 |
| 关闭标签追踪 | `label_track_off` | 调用 `/label_track/exit` 服务 |
| 舵机打开 | `uart_open` | 调用 `/servo/enter` 服务 |
| 舵机关闭 | `uart_close` | 调用 `/servo/exit` 服务 |
| JSON 字符串 | `{"id":1,"angle":90}` | 发布到 `joint_commands` topic |

### 1.4 离线系统特点

| 特性 | 说明 |
|------|------|
| **优势** | 零延迟、无需网络、稳定可靠 |
| **劣势** | 必须说精确命令词（"开启颜色分拣"≠"开始分拣颜色"） |
| **词汇量** | 约 40 条固定命令词 |
| **扩展方式** | 修改 ASRPRO 固件 + 修改 asr_node.py 代码 |

## 二、在线语音交互方案设计

### 2.1 核心思路

**用 LLM 理解用户意图，将自然语言映射到与离线系统相同的 ROS2 服务/Topic。**

```
用户说 "帮我把红色的挑出来"          用户说 "开启颜色分拣"
        │                                    │
        ▼                                    ▼
   ASR 识别文本                          ASR 识别文本
   "帮我把红色的挑出来"                  "开启颜色分拣"
        │                                    │
        ▼                                    ▼
   LLM 语义理解                          LLM 语义理解
   → 颜色分拣 + 红色                    → 颜色分拣
        │                                    │
        ▼                                    ▼
   输出: color_sorting_enter             输出: color_sorting_enter
         + sort red
        │                                    │
        └────────────┬───────────────────────┘
                     ▼
            asr_node.py 同样的分发逻辑
            调用同样的 ROS2 服务/Topic
```

### 2.2 两种实现路径

#### 路径 A：LLM 直接输出 ROS2 服务调用（推荐）

```
voice_recognition_node  →  intent_parser_node  →  执行器节点
     (ASR)                   (LLM 语义理解)         (调用 ROS2 服务)
```

**LLM 的任务：** 将任意自然语言映射为标准化的 JSON 指令。

**执行器节点的任务：** 解析 JSON，调用对应的 ROS2 服务/Topic。

**优势：**
- LLM 只负责"理解"，执行器只负责"执行"，职责清晰
- 修改命令只需改 SYSTEM_PROMPT，不改代码
- 可以处理复合指令（"先开启颜色分拣，再分拣红色"）

#### 路径 B：LLM 输出离线系统兼容字符串

```
voice_recognition_node  →  intent_parser_node  →  asr_node.py
     (ASR)                   (LLM → 输出字符串)      (已有分发逻辑)
```

**LLM 的任务：** 将自然语言翻译为离线系统已有的命令字符串。

**优势：** 复用 asr_node.py，不写新代码。
**劣势：** 受限于离线系统的命令粒度，无法表达更灵活的指令。

**结论：推荐路径 A，扩展性更强。**

---

### 2.3 路径 A 详细方案

#### 2.3.1 系统架构

```
┌─────────────────────┐     voice_text      ┌──────────────────────┐
│ voice_recognition   │ ──── (String) ────> │   intent_parser      │
│ _node               │                     │   _node              │
│                     │                     │                      │
│ 阿里云 ASR          │                     │ qwen-plus LLM        │
│ 流式语音识别        │                     │ 自然语言→JSON        │
└─────────────────────┘                     └──────┬───────┬───────┘
                                                    │       │
                                          voice_command   speak_text
                                           (String)       (String)
                                                │               │
                                                ▼               ▼
                                    ┌──────────────┐  ┌──────────────────┐
                                    │ arm_executor │  │ voice_synthesis  │
                                    │   _node      │  │ _node            │
                                    │              │  │                  │
                                    │ 解析JSON     │  │ qwen-tts 合成    │
                                    │ 调用ROS2服务  │  │ 声卡播放         │
                                    └──────────────┘  └──────────────────┘
```

#### 2.3.2 LLM SYSTEM_PROMPT 设计

SYSTEM_PROMPT 是整个方案的核心，它决定了 LLM 能理解哪些指令、输出什么格式。

**设计原则：**
1. 列出所有支持的功能（对应离线系统的全部命令）
2. 每个功能有明确的 function 名和参数格式
3. 用户可以用任意自然语言表达，LLM 负责语义匹配

**功能清单（对应 asr_node.py 的全部动作）：**

```
功能分类          function 名              参数
──────────────────────────────────────────────────
颜色分拣          color_sorting            enter / exit / colors: [red,green,blue]
颜色码垛          color_stack              enter / exit
颜色追踪          color_track              enter / exit / color: red|green|blue
标签分拣          label_sorting            enter / exit / labels: [1,2,3]
标签码垛          label_stack              enter / exit
标签追踪          label_track              enter / exit / label: 1|2|3
数字分拣          num_sorting              enter / exit / nums: [1,2,3]
数字码垛          num_stack                enter / exit
数字追踪          num_track                enter / exit / num: 1|2|3
颜色设置          color_set                enter / exit
人脸追踪          face_track               enter / exit
舵机控制          servo                    enter / exit
关节控制          joint                    id: 1-6, angle: -90~90
固定动作          routine                  action: 夹爪开|夹爪关|复位|...
```

**SYSTEM_PROMPT 示例结构：**

```
你是一个桌面6轴机械臂语音助手，同时控制视觉系统。
注意：LLM 不需要知道相机类型，执行器节点会自动根据接入的相机路由到对应的节点。

回复格式：
1. {"message": "回复提示"}
2. {"step": {"order": 1, "function": "<功能>", "parameters": {<参数>}}}

支持的功能：

颜色分拣:
  - function: "color_sorting"
  - parameters: {"action": "enter"} 或 {"action": "exit"} 或 {"action": "sort", "colors": ["red"]}
  - 用户说"开启颜色分拣"、"开始分拣颜色"、"帮我把红色的挑出来" → 都映射到此功能

颜色追踪:
  - function: "color_track"
  - parameters: {"action": "enter"} 或 {"action": "exit"} 或 {"action": "track", "color": "red"}
  - 用户说"追踪红色"、"跟着红色走"、"盯住红色的东西" → 都映射到此功能

... (以此类推)
```

#### 2.3.3 执行器节点 (arm_executor_node)

**职责：** 订阅 `/voice_command`，解析 JSON，调用对应的 ROS2 服务/Topic。

**相机检测：** 复用 asr_node.py 中已实现的 `detect_camera()` 函数，启动时检测 USB 相机类型，自动路由到 2D 或深度节点。

```python
from asr_node import detect_camera  # 复用离线节点的检测函数

class ArmExecutorNode(Node):
    def __init__(self):
        self.camera_mode = detect_camera()  # "depth" 或 "mono"
        self.prefix = 'depth_' if self.camera_mode == 'depth' else ''

    def execute_step(self, step):
        func = step["function"]
        params = step.get("parameters", {})

        if func == "color_sorting":
            if params["action"] == "enter":
                self.call_service(f'/{self.prefix}color_sorting/enter')
            elif params["action"] == "exit":
                self.call_service(f'/{self.prefix}color_sorting/exit')
            elif params["action"] == "sort":
                self.publish(f'/{self.prefix}color_sorting/command',
                             ",".join(params["colors"]))

        elif func == "color_track":
            if params["action"] == "enter":
                self.call_service(f'/{self.prefix}color_track/enter')
            elif params["action"] == "track":
                self.publish('/color', params["color"])  # 追踪目标 Topic 共用

        elif func == "joint":
            self.publish('joint_commands', json.dumps(params))

        # ... 其他功能同理，分拣/码垛/追踪用 self.prefix，其余不变
```

**路由规则：**
- 分拣/码垛/追踪的 enter/exit + command → 使用 `self.prefix`
- 追踪目标 (`/color`, `/label`, `/num`) → 共用 Topic，不加前缀
- color_set、face_track、servo、关节控制 → 无前缀（与相机无关）

#### 2.3.4 灵活性对比

| 用户说法 | 离线系统 | 在线系统 |
|---------|---------|---------|
| "开启颜色分拣" | ✅ 命中 | ✅ 命中 |
| "开始分拣颜色" | ❌ 未命中 | ✅ 命中 (语义相同) |
| "帮我把红色的挑出来" | ❌ 未命中 | ✅ 命中 (理解意图) |
| "把红色和蓝色分拣一下" | ❌ 未命中 | ✅ 命中 (复合动作) |
| "先开启分拣，然后分拣红色" | ❌ 未命中 | ✅ 命中 (多步指令) |
| "不要了，停" | ❌ 未命中 | ✅ 命中 (理解"停"=退出) |
| "那个蓝色的帮我追踪一下" | ❌ 未命中 | ✅ 命中 (口语化表达) |

### 2.4 与现有系统的关系

#### 2.4.1 共存方案

离线和在线系统**可以共存**，通过一个模式切换节点管理：

```
模式切换方式：
1. 硬件按钮切换
2. 语音命令切换（如说 "切换到在线模式"）
3. ROS2 服务切换（/voice_mode/switch）

当前模式 → asr_node.py (离线)
当前模式 → voice_interaction + arm_executor_node (在线)
```

#### 2.4.2 Topic 接口对齐

在线系统的执行器节点必须发布到与离线系统**完全相同的 Topic/Service**，才能驱动相同的视觉/机械臂节点。执行器节点复用 asr_node.py 的 `detect_camera()` 函数自动选择路径：

| 动作 | 离线系统（2D 模式） | 离线系统（深度模式） | 在线系统 |
|------|-------------------|-------------------|---------|
| 颜色分拣进入 | `call /color_sorting/enter` | `call /depth_color_sorting/enter` | 自动选择 |
| 颜色分拣指令 | `publish /color_sorting/command` | `publish /depth_color_sorting/command` | 自动选择 |
| 颜色追踪进入 | `call /color_track/enter` | `call /depth_color_track/enter` | 自动选择 |
| 追踪颜色 | `publish /color` | `publish /color` (共用) | 相同 |
| 关节控制 | `publish joint_commands` | `publish joint_commands` | 相同 |
| 舵机开关 | `call /servo/enter` | `call /servo/enter` | 相同 |

## 三、实现步骤

### 3.1 需要新建的文件

| 文件 | 说明 |
|------|------|
| `arm_executor_node.py` | 新节点：订阅 voice_command，调用 ROS2 服务/Topic |
| 更新 `intent_parser_node.py` | SYSTEM_PROMPT 改为覆盖全部功能 |
| 更新 `voice_params.yaml` | 添加执行器节点参数 |
| 更新 `voice_interaction_launch.py` | 添加执行器节点 |

### 3.2 需要修改的 SYSTEM_PROMPT

当前 SYSTEM_PROMPT 只覆盖了机械臂基础动作（grasp/place/joint/routine/home/stop），需要扩展为覆盖全部 2D 视觉功能：

```
当前覆盖:
  ✅ grasp (抓取)
  ✅ place (放置)
  ✅ joint (关节控制)
  ✅ routine (固定动作)
  ✅ home (复位)
  ✅ stop (停止)

需要新增:
  ❌ color_sorting (颜色分拣: enter/exit/sort)
  ❌ color_stack (颜色码垛: enter/exit)
  ❌ color_track (颜色追踪: enter/exit/track)
  ❌ label_sorting (标签分拣: enter/exit/sort)
  ❌ label_stack (标签码垛: enter/exit)
  ❌ label_track (标签追踪: enter/exit/track)
  ❌ num_sorting (数字分拣: enter/exit/sort)
  ❌ num_stack (数字码垛: enter/exit)
  ❌ num_track (数字追踪: enter/exit/track)
  ❌ color_set (颜色设置: enter/exit)
  ❌ face_track (人脸追踪: enter/exit)
  ❌ servo (舵机控制: enter/exit)
```

### 3.3 执行器节点需对接的 ROS2 接口

从 asr_node.py 提取的完整 ROS2 接口清单（执行器节点根据 `self.prefix` 自动选择 2D 或深度路径）：

```

  2D:    /color_sorting/enter    深度: /depth_color_sorting/enter
  2D:    /color_sorting/exit     深度: /depth_color_sorting/exit
  2D:    /color_stack/enter      深度: /depth_color_stack/enter
  2D:    /color_stack/exit       深度: /depth_color_stack/exit
  2D:    /color_track/enter      深度: /depth_color_track/enter
  2D:    /color_track/exit       深度: /depth_color_track/exit
  2D:    /label_sorting/enter    深度: /depth_label_sorting/enter
  2D:    /label_sorting/exit     深度: /depth_label_sorting/exit
  2D:    /label_stack/enter      深度: /depth_label_stack/enter
  2D:    /label_stack/exit       深度: /depth_label_stack/exit
  2D:    /label_track/enter      深度: /depth_label_track/enter
  2D:    /label_track/exit       深度: /depth_label_track/exit
  2D:    /num_sorting/enter      深度: /depth_num_sorting/enter
  2D:    /num_sorting/exit       深度: /depth_num_sorting/exit
  2D:    /num_stack/enter        深度: /depth_num_stack/enter

  2D:    /num_track/enter        深度: /depth_num_track/enter
  2D:    /num_track/exit         深度: /depth_num_track/exit

Services (无前缀，与相机无关):
  /color_set/enter, /color_set/exit
  /face_track/enter, /face_track/exit
  /servo/enter, /servo/exit

Topics (路径 = /{prefix}xxx/command):
  2D:    /color_sorting/command       深度: /depth_color_sorting/command
  2D:    /label_sorting/sort_command  深度: /depth_label_sorting/sort_command

Topics (无前缀):
  joint_commands  ← JSON: {"id":1,"angle":90}
```

## 四、延迟与成本

### 4.1 端到端延迟

| 阶段 | 延迟 |
|------|------|
| ASR (语音→文本) | ~1-2s |
| LLM (文本→指令) | ~1-3s |
| TTS (文本→语音) | ~1-2s (与 LLM 并行) |
| 执行器 (指令→ROS2) | <10ms |
| **总计** | **~3-5s** |

### 4.2 API 成本

| 服务 | 计费 | 估算单次成本 |
|------|------|-------------|
| ASR (Paraformer) | 按音频时长 | ~¥0.006/次 (3秒语音) |
| LLM (qwen-plus) | 按 token | ~¥0.002/次 (200 token) |
| TTS (qwen-tts) | 按字符数 | ~¥0.01/次 (20字符) |
| **单次交互** | | **~¥0.02** |

## 五、总结

| 维度 | 离线系统 (ASRPRO) | 在线系统 (voice_interaction) |
|------|------------------|---------------------------|
| **识别方式** | 硬件命令词匹配 | 云端 ASR + LLM 语义理解 |
| **灵活性** | 必须说精确命令词 | 任意自然语言表达 |
| **延迟** | <0.5s | ~3-5s |
| **网络依赖** | 无 | 必须联网 |
| **扩展方式** | 改固件+改代码 | 改 SYSTEM_PROMPT |
| **成本** | 零 | ~¥0.02/次 |
| **适用场景** | 固定流程操作 | 灵活交互、教学演示 |

**推荐方案：路径 A（LLM 直接输出 JSON + 执行器节点调用 ROS2 服务）**
**与离线系统共存，通过模式切换选择使用哪套系统。**
