## 一、系统概述

### 1.1 定位

基于阿里云 DashScope 的**全在线**语音交互 ROS2 功能包，实现"语音输入 → 自然语言理解 → 结构化指令输出 → 语音回复"的完整管线。面向桌面 6 轴机械臂场景。

### 1.2 技术栈

| 组件 | 模型/方案 | 提供方 | 运行位置 |
|------|----------|--------|---------|
| 语音识别 (ASR) | Paraformer-realtime-v2 | 阿里云 DashScope | 云端流式 |
| 自然语言理解 (LLM) | qwen-plus | 通义千问 (OpenAI 兼容 API) | 云端 |
| 语音合成 (TTS) | qwen-tts | 阿里云 DashScope | 云端 |

### 1.3 依赖

**Python 包：**
- `dashscope` >= 1.25 — ASR (Recognition) + TTS (qwen_tts.SpeechSynthesizer)
- `openai` >= 2.0 — LLM (OpenAI 兼容接口)
- `numpy` — 音频重采样

**系统工具：**
- `arecord` (alsa-utils) — 麦克风音频采集
- `aplay` (alsa-utils) — 音频播放

**ROS2：**
- `rclpy` / `std_msgs` — 节点框架与消息传递

## 二、整体架构

### 2.1 节点拓扑

```
┌─────────────────────┐     voice_text      ┌──────────────────────┐
│ voice_recognition   │ ──── (String) ────> │   intent_parser      │
│ _node               │                     │   _node              │
│                     │                     │                      │
│ 阿里云 ASR          │                     │ qwen-plus LLM        │
│ 流式语音识别        │                     │ 自然语言→JSON        │
└─────────────────────┘                     └──────┬───────┬───────┘
                                                    │       │
                                          voice_command   speak_text
                                           (String)       (String)
                                                │       │
                                                ▼       ▼
                                    ┌──────────────┐ ┌──────────────────┐
                                    │  执行器节点   │ │ voice_synthesis  │
                                    │  (待实现)     │ │ _node            │
                                    │              │ │                  │
                                    │ 驱动机械臂   │ │ qwen-tts 合成    │
                                    └──────────────┘ └──────────────────┘
```

### 2.2 Topic 数据流

| Topic | 类型 | 发布者 | 订阅者 | 内容 |
|-------|------|--------|--------|------|
| `voice_text` | `std_msgs/String` | voice_recognition_node | intent_parser_node | ASR 识别文本 |
| `voice_command` | `std_msgs/String` | intent_parser_node | 外部执行器 | JSON 结构化指令 |
| `speak_text` | `std_msgs/String` | intent_parser_node | voice_synthesis_node | TTS 待合成文本 |

### 2.3 完整数据流路径

```
1. 用户说话
   │
   ▼
2. arecord 采集 48kHz PCM 音频 (S16_LE, 单声道)
   │
   ▼
3. numpy 降采样 48kHz → 16kHz (取每3个采样点的第1个)
   │
   ▼
4. dashscope Recognition 流式发送音频帧到阿里云
   │  ASRCallback.on_event() 回调
   │  检测句子结束标志 (is_sentence_end 或标点符号)
   │
   ▼
5. 识别结果放入 result_queue
   │  _poll_result() 定时器 (100ms) 轮询
   │
   ▼
6. 发布到 /voice_text
   │
   ▼
7. intent_parser_node 收到文本
   │  启动异步线程调用 qwen-plus API (stream=True)
   │
   ▼
8. 流式解析 LLM 输出
   │  ├─ 提取 "message" 字段 → 发布到 /speak_text → TTS 播报
   │  └─ 提取 "step" JSON → 发布到 /voice_command → 执行器
   │
   ▼
9. voice_synthesis_node 收到 speak_text
   │  调用 dashscope qwen-tts API
   │  获取音频 URL → 下载 WAV → aplay 播放
   │
   ▼
10. 用户听到语音回复
```

## 三、节点详细分析

### 3.1 voice_recognition_node — 语音识别节点

**文件：** `voice_interaction/voice_recognition_node.py` (179 行)

**类结构：**

```
ASRCallback (RecognitionCallback)     VoiceRecognitionNode (Node)
├── on_event(result)                  ├── __init__()
│   ├── 解析 sentence.text            │   ├── 声明5个参数
│   ├── 检测句子结束标志              │   ├── 创建 publisher (voice_text)
│   └── 放入 result_queue             │   ├── 创建 timer (100ms)
├── on_error(result)                  │   └── 启动 _listen_loop 线程
└── on_complete()                     ├── _listen_loop()  [后台线程]
                                      │   ├── 检测麦克风设备
                                      │   ├── 启动 arecord 子进程
                                      │   ├── 创建 Recognition 实例
                                      │   ├── 循环读取+降采样+发送
                                      │   └── 异常自动重连
                                      ├── _poll_result()  [定时器回调]
                                      │   └── 从 queue 取结果发布
                                      └── destroy_node()
```

**关键参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `api_key` | (空) | DashScope API Key，空则读 `DASHSCOPE_API_KEY` 环境变量 |
| `asr_model` | `paraformer-realtime-v2` | 阿里云端到端语音识别模型 |
| `audio_device` | `default` | ALSA 录音设备名 |
| `record_rate` | `48000` | 实际录音采样率 (Hz) |
| `asr_rate` | `16000` | ASR 要求的采样率 (Hz) |

**音频处理细节：**

```
录音参数: 48kHz / S16_LE / 1声道
每次读取: 9600 采样点 × 2字节 = 19200 字节 (200ms 音频)
降采样:   48000 / 16000 = 3:1，取每第3个采样点
发送帧:   3200 采样点 × 2字节 = 6400 字节 (200ms @ 16kHz)
```

**容错机制：**
- 麦克风设备不可用：等 5 秒重试
- arecord 进程异常退出：自动重启监听循环
- ASR 连接出错：等 3 秒重连
- 超时检测：麦克风检测 5 秒超时

**ASRCallback 句子结束判定：**
1. ASR 返回 `is_sentence_end=True`（语音端点检测）
2. 文本末尾是标点符号：`。？！?!.~`

### 3.2 intent_parser_node — 意图解析节点

**文件：** `voice_interaction/intent_parser_node.py` (180 行)

**类结构：**

```
IntentParserNode (Node)
├── __init__()
│   ├── 声明3个参数 (api_key, base_url, model)
│   ├── 创建 OpenAI 客户端
│   ├── 订阅 voice_text
│   └── 发布 voice_command + speak_text
├── text_callback(msg)
│   └── 启动 _parse_async 线程
├── _parse_async(text)  [异步线程]
│   ├── 调用 LLM (stream=True)
│   ├── 流式提取 "message" → 发布 speak_text
│   └── 流式提取 "step" JSON → 发布 voice_command
└── _extract_steps(buffer)
    ├── 大括号深度匹配提取完整 JSON
    ├── 验证 JSON 合法性
    └── 返回未处理完的 buffer
```

**关键参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `api_key` | (空) | DashScope API Key |
| `base_url` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | OpenAI 兼容端点 |
| `model` | `qwen-plus` | LLM 模型名 |

**SYSTEM_PROMPT 设计要点：**

```
角色: 桌面6轴机械臂语音助手
输出格式:
  1. {"message": "回复提示"}  ← 给用户听的自然语言反馈
  2. {"step": {              ← 给执行器的结构化指令
       "order": 1,
       "function": "<功能>",
       "parameters": {<参数>}
     }}

支持的 function:
  - grasp (抓取)     → {"object": "物体描述"}
  - place (放置)     → {"location": "位置描述"}
  - joint (关节控制) → {"id": 1-6, "angle": 角度}
  - routine (固定动作) → {"action": "动作名"}
  - home (复位)
  - stop (停止)
```

**流式 JSON 解析算法 (`_extract_steps`)：**

```
输入: buffer (LLM 流式输出的累积文本)

循环:
  1. 找到第一个 '{'
  2. 大括号深度计数: '{' +1, '}' -1
  3. 深度归零时找到完整 JSON 边界
  4. json.loads() 解析
  5. 若含 "step" 键 → 发布到 voice_command
  6. 移除已处理部分，继续找下一个

返回: 未处理完的尾部 buffer (可能是不完整的 JSON)
```

**并发模型：**
- 主线程：ROS2 spin 处理回调
- 每次收到 voice_text 启动一个 daemon 线程调用 LLM
- 多条指令可并发处理（不阻塞）

### 3.3 voice_synthesis_node — 语音合成节点

**文件：** `voice_interaction/voice_synthesis_node.py` (84 行)

**类结构：**

```
VoiceSynthesisNode (Node)
├── __init__()
│   ├── 声明3个参数 (api_key, voice, audio_device)
│   ├── 设置临时文件路径 /tmp/tts_output.wav
│   └── 订阅 speak_text
├── speak_callback(msg)
│   └── 启动 _speak 线程
└── _speak(text)  [异步线程]
    ├── SpeechSynthesizer.call(model, text, voice)
    ├── 获取音频 URL
    ├── urllib 下载 WAV
    └── aplay 播放
```

**关键参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `api_key` | (空) | DashScope API Key |
| `voice` | `Cherry` | 音色: Cherry / Ethan / Serena / Chelsie |
| `audio_device` | `default` | ALSA 播放设备名 |

**qwen-tts API 调用细节：**

```python
from dashscope.audio.qwen_tts import SpeechSynthesizer  # 注意：不是 dashscope.audio.tts

response = SpeechSynthesizer.call(
    model='qwen-tts',      # 固定模型名
    text='你好',            # 待合成文本
    voice='Cherry',         # 音色选择
)
# 返回 TextToSpeechResponse
# response["output"]["audio"]["url"] → WAV 文件 URL (24kHz, S16_LE, 单声道)
```

**音频播放流程：**
1. 下载 WAV 到 `/tmp/tts_output.wav`
2. `aplay -D default /tmp/tts_output.wav` 播放
3. WAV 格式：24000Hz, 16bit, 单声道（aplay 自动识别头信息）

## 七、与 src_25.12.29 car_llm 系统对比

| 维度 | car_llm (src_25.12.29) | voice_interaction (本包) |
|------|----------------------|------------------------|
| **架构** | 多进程 + Queue | 多节点 + ROS2 Topic |
| **ASR** | PyArmor 加密封装 | dashscope SDK 直接调用 |
| **LLM** | PyArmor 加密封装 | OpenAI SDK 直接调用 |
| **TTS** | PyArmor 加密封装 (qwen-tts) | dashscope SDK 直接调用 (qwen-tts) |
| **硬件依赖** | AIBox 语音盒子 (USB 复合设备) | 任意 USB 声卡 / 系统默认设备 |
| **唤醒方式** | AIBox 硬件唤醒词 | 无（持续监听） |
| **机械臂控制** | 内置 ros_control 节点 | 无（通过 voice_command Topic 对接） |
| **底盘控制** | 支持 /cmd_vel | 无（纯机械臂） |
| **LLM Prompt** | 3 套 YAML (car/arm/c2) | 1 套内嵌 Prompt (纯机械臂) |
| **进程间通信** | multiprocessing.Queue | ROS2 Topic (std_msgs/String) |
| **扩展性** | 修改 YAML Prompt | 修改 SYSTEM_PROMPT 或添加新节点 |

**核心区别：** 本包将 car_llm 的单体多进程架构拆分为标准 ROS2 多节点架构，每个功能独立可替换，通过 Topic 解耦。

## 九、已知限制与改进方向

### 9.1 当前限制

| 问题 | 说明 |
|------|------|
| **无唤醒机制** | 持续监听，无法区分"对机器人说话"和"日常对话" |
| **无上下文记忆** | 每次 LLM 调用独立，不保留对话历史 |
| **无执行器节点** | voice_command 发布后无人消费，机械臂不动作 |
| **无并发控制** | 多条语音指令可能同时触发 LLM 和 TTS |
| **TTS 阻塞播放** | aplay 是同步调用，播放期间新的 TTS 请求排队 |
| **单声道固定** | 录音和播放参数硬编码，不支持多声道设备 |
| **无离线降级** | 网络断开时完全不可用 |

### 9.2 改进方向

| 方向 | 方案 |
|------|------|
| 唤醒词 | 集成 Porcupine / Snowboy 本地唤醒，或复用 AIBox 硬件唤醒 |
| 对话上下文 | intent_parser_node 维护 messages 列表，支持多轮对话 |
| 执行器节点 | 新建 arm_executor_node 订阅 voice_command，调用 IK SDK 驱动机械臂 |
| 并发队列 | TTS 节点内部维护播放队列，串行播放 |
| 离线降级 | ASR 切换为 Vosk (本地)，LLM 切换为 llama.cpp (本地) |
| 多语言 | ASR 已支持 language_hints=['zh', 'en']，LLM Prompt 可扩展 |

### 全链路延迟估算

| 阶段 | 延迟 |
|------|------|
| ASR (语音→文本) | ~1-2s (流式，句子结束时返回) |
| LLM (文本→指令) | ~1-3s (流式，首个 chunk 约 0.5s) |
| TTS (文本→语音) | ~1-2s (API 调用 + 下载 + 播放) |
| **端到端** | **~3-7s** |
